## Real bugs found and fixed during this UAT

### Bug #1 — Stripe SDK 15.x metadata access broken (commits c50d7d4 + 65c8a19)

**Symptom**: AC16 redirect chain failed with `?stripe_error=1`. Server log: `stripe_connect_return: failed to retrieve account ...: get`.

**Root cause**: Stripe SDK 15.1 exposes `Account.metadata` as a `StripeObject` whose attribute access for `.get` raises `AttributeError`. Three patterns tested:
- `acct.metadata.get("user_id")` → `AttributeError: 'get'` (the original code)
- `dict(acct.metadata).get("user_id")` → `KeyError: 0` (StripeObject lacks proper dict iteration)
- `acct.metadata["user_id"]` → returns `'27'` ✅ (the only access pattern that works)

**Fix applied to two sites**:
- `server/app/routers/user/stripe.py:88` — Connect onboarding return URL (was breaking AC16 entirely)
- `server/app/services/payments.py:86` — checkout.session.completed webhook (would have broken AC4 — company balance crediting at scale)

**Tests**: 336 pytest still green after fix.

## Strategic blocker discovered (NOT a UAT failure — product-level constraint)

**Stripe platform `acct_1TCGfuABBUrjm7YF` (Stock Buddy sandbox) is registered in country=AE (UAE).**

Stripe enforces a geo-restriction: **AE platforms cannot create US-country Connect accounts**. Verified directly via the Stripe API:
```
POST /v1/accounts { type=express, country=US, ... }
→ "Connected accounts in US cannot be created by platforms in AE."
```

When `country` is omitted, Stripe creates a Connect account in the platform's own country (AE) by default — which is what happened in AC15 (`acct_1TSghiAhD4HudVVs.country == "AE"`). This works for the cascading-chain UAT but is **strategically incorrect for the US-only target audience**.

**Implication for v1.0.0 launch**: Before public launch, the platform must move to a US-based Stripe entity (e.g., via Stripe Atlas or by registering a US C-corp). This is the same checklist item as the existing pre-launch `live keys swap` (per `docs/migrations/2026-05-01-migration-gap-audit.md`).
