## Notes

- The original Task #86 fix added one resolver flag (`AsyncDns`). Task #88 layers three more flag families addressing distinct DNS regressions in modern Chromium: DoH (Chromium-managed secure DNS that some networks block silently), Happy Eyeballs V3 (experimental dual-stack racing path that fails when one stack lies dead), and the intermediate-certificate verifier updater (a slow background fetch on cold start that can stall first navigation).
- Two new escape-hatch env vars give users on broken networks an out without re-deploying:
  - `AMPLIFIER_BROWSER_DNS_PREFER_IPV4=1` — for users whose IPv6 stack is half-up
  - `AMPLIFIER_BROWSER_DNS_DOH_SERVER=<url>` — pin DoH to a known-good resolver (Cloudflare 1.1.1.1, Google 8.8.8.8, etc.)
- Optional fresh-laptop smoke (NOT required for marking done): on a Jio-IPv6 network, `python scripts/login_setup.py linkedin` should now navigate cleanly. This is environment-dependent — captured as a regression-test recipe for the maintainer in `docs/specs/dns-robustness.md`.
- A new helper script `scripts/uat/probe_dns_hardening.py` (~50 LOC) launches Patchright with the production hardening args and verifies clean startup. Reusable for future DNS-related regressions.
