## New product bugs found this run

### Bug A — `get_user_profiles()` reads wrong table (FIXED, tracked as Task #55)
- `scripts/utils/local_db.py:956` was reading `agent_user_profile` (always empty) instead of `scraped_profile` (where the profile scraper actually writes)
- Impact: `_refine_strategy_with_ai()` received `user_profiles=[]` for every user since 2026-04-18. AC5 had been silently no-op'ing the entire AI strategy refinement layer.
- **Fixed in this run**. `agent_user_profile` table dropped, `get_user_profiles()` now reads from `scraped_profile` with column adapter. Verified: returns 3 profiles for the UAT user with correct LinkedIn follower count (9494).

### Bug A.1 — `'list' object has no attribute 'get'` in content agent
- After Bug A was fixed and real profile data started flowing, the content agent immediately hit `'list' object has no attribute 'get'` errors
- I added validation in `_refine_strategy_with_ai` to reject AI-returned strategies where `platforms` is a list instead of a dict
- **The bug PERSISTS even after the fix** — visible on the dashboard with timestamps `2026-04-26 05:09:40`, `05:00:50`, `04:58:12` AFTER my fix and module reload
- Root cause is somewhere else; my fix protects one path but the error still reaches the activity feed
- **Status: NOT fully fixed.** Add as Task #56.

### Bug C — `seed_campaign.py` tries to upload images to a non-existent endpoint
- Script POSTs to `/api/storage/upload` (returns 404)
- Actual endpoint: `/company/campaigns/upload-asset` (uses cookie auth, not Bearer token)
- Impact: AC3 (vision analysis) cannot run — campaign has no images
- **Workaround applied**: skip image upload for this UAT
- **Status: needs proper fix** — script should authenticate via the company HTML form to get a cookie session, then upload via the correct endpoint

### Bug D — Quality gate (Task #15) blocks UAT seed campaigns
- `seed_campaign.py` defaults (CPM $0.50, no niche tags, no required platforms) score 64/100; threshold is 85
- This is good news: Task #15's quality gate works correctly!
- **Workaround applied**: bumped seed defaults to CPM $2, added niche_tags + required_platforms, budget $150
- **Status: workaround sufficient.** Captured as a learning in `docs/uat/skills/uat-task/LEARNINGS.md` so future seed runs default correctly.

### Bug E — Quality gate counts `target_regions=[]` as "no targeting"
- The rubric expects at least one of: niche tags, required platforms, target regions
- An empty list satisfies "I'm targeting everywhere" semantically but fails the rubric
- **Workaround applied** via Bug D's fix (added niche_tags + required_platforms; left target_regions empty)
- Per the user's correction, this is by-design: per-campaign targeting decides who matches; empty regions = match anywhere. So the rubric should accept this combination.
- **Status: minor rubric refinement needed in Task #15 implementation.** Add as Task #57.

### Bug F — Matching algorithm did not invite the test user to an active campaign
- Test user has 9.4k LinkedIn followers, niches `["financial markets", "algorithmic trading", "software development"]`, region detected
- Test campaign 15 is active, niche_tags `["trading","finance","investing"]`, no region restriction, $150 budget, CPM $2
- After 30 seconds of polling `/api/campaigns/mine`, no invitation appeared
- Likely root causes: (a) Bug #53 zero follower counts on Reddit/Facebook causing `min_followers` filter to reject (b) something in the matching service. Need server-side investigation.
- **Status: blocking. Add as Task #58.** This is a money-loop killer.

### Bug G — Duplicate invitation rendered on /campaigns page
- Two cards titled "UAT Test Campaign" both linking to /campaigns/7 — same campaign rendered twice
- Could be a DB duplicate (server-side dup invitation row) OR a UI rendering bug (frontend de-dup missing)
- **Status: not investigated. Add as Task #59.**

### Existing Bug — X shows "Connected" on dashboard despite global disable (Task #40)
- The platform health card on the dashboard renders X as "Connected"
- X is system-disabled per Task #40, no posting/scraping happens
- The dashboard UI doesn't reflect this — it should show X as "Disabled" with an explanation, or hide it entirely
