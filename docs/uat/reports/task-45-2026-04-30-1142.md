## Bugs surfaced + fixed during UAT

1. **Initial agent attempt would have produced an empty baseline** because the plan called for `init_tables()` (create_all) before autogenerate, which would yield zero diffs. Corrected: autogenerate runs against an EMPTY DB first, capturing all tables as `op.create_table(...)` calls.
2. **`pg_dump` and `psql` not on Windows PATH** — UAT helpers route through `docker compose exec -T postgres` instead.
3. **AC2's `_create_all()` path required `importlib.reload(app.core.database)`** because the module-level engine is created at import time; setting `DATABASE_URL` AFTER first import doesn't take effect on the cached engine.
4. **`alembic current` outputs nothing on Windows + Python 3.14** when the revision IS at head (just INFO logs to stderr, empty stdout). AC3 originally asserted `"(head)" in output` which failed despite stamp being correct. Fixed: AC3 now queries `alembic_version` table directly via asyncpg — that's the actual contract.
5. **VPS `amplifier` venv missing `psycopg2-binary`** — alembic env.py converts `+asyncpg` URL to `+psycopg2` for migrations (sync). Installed `psycopg2-binary 2.9.12` in the prod venv.
6. **`alembic` couldn't find `app.models`** when run from repo root — env.py imports `from app.models import *` which requires running from `server/` cwd. UAT helper sets cwd correctly.

## Note on JSON vs JSONB

Documented in CLAUDE.md and worth flagging here:

- Models declare columns as `from sqlalchemy import JSON as JSONB` — a portability alias for SQLite + Postgres.
- Baseline migration emits `sa.JSON()` (which Postgres stores as `json` text type).
- Production was manually patched during Task #41 to use native `jsonb` (per `docs/migrations/2026-04-25-task41-schema-fixes.md`).
- `alembic check` against the prod DB currently shows zero diffs (SQLAlchemy's `JSON` type is treated as compatible with both `json` and `jsonb` for diff purposes).
- If any future query needs `jsonb`-only features (GIN indexes, `@>` containment, jsonb_path_ops), file a follow-up migration using `sa.dialects.postgresql.JSONB` and run against prod. Not blocking.

## Going-forward enforcement

CLAUDE.md now states:

> Every PR that changes a SQLAlchemy model in `server/app/models/` MUST include a corresponding Alembic migration in `server/alembic/versions/`. No exceptions.

When the next model change happens (e.g. Task #19 Stripe live integration, or BYOK Task #70):
1. Edit model
2. `cd server && alembic revision --autogenerate -m "describe_change"` against a fresh test DB
3. Review diff
4. Test locally with `alembic upgrade head`
5. Commit BOTH model and migration
6. Deploy: SSH, `alembic upgrade head`, restart `amplifier-web.service`

This unblocks Phase D migrations (#66/#67/#68) — they can introduce schema changes safely.
