  ---
  Now the branch-vs-repo question. Here's the CTO call.

  Don't create a new repo. Don't create a new branch yet either.

  Here's why.

  What you already have

  The original Amplifier engine is still fully intact in this repo:
  - scripts/post.py — posts to LinkedIn (+ 5 other platforms) via Playwright
  - scripts/engine/ — declarative JSON posting engine
  - config/scripts/linkedin_post.json — LinkedIn-specific selectors
  - scripts/generate.ps1 — content generation via Claude CLI
  - scripts/review_dashboard.py — draft review UI
  - scripts/ai/ — Gemini/Mistral/Groq providers with auto-fallback
  - scripts/utils/image_manager.py — image pipeline with UGC post-processing

  That IS the personal LinkedIn bot. It was never removed. The marketplace (server/, user app) was added on top, not
  instead of.

  What you DON'T need

  - A new repo → 90% code duplication. AI providers, posting engine, image pipeline, JSON scripts, human_behavior,
  review dashboard all have to be copied or submoduled. Two repos to maintain. Bug fixes drift. Don't do it.
  - A new branch right now → premature. You haven't even changed anything yet. Branch when you start making code
  changes that would break the marketplace codebase — not before.

  What you actually do

  1. Ship the profile rewrite first — that's pure LinkedIn UI work. Zero code. No branch needed.
  2. Then restart content, which you can already do with powershell scripts/generate.ps1 + the review dashboard. Turn
   off the platforms you don't want in config/platforms.json (keep only LinkedIn enabled). Still no code changes.
  3. Only when you need to make LinkedIn-specific improvements to the bot (custom LI-only content generator, a LI
  engagement agent, etc.) → then branch off main as personal-li-bot.

  The one rule

  Do not work from flask-user-app (your current branch). That's marketplace work. When the time comes, check out main
   first, then branch from there.

  When a separate repo WOULD make sense

  Only if (a) you open-source the personal bot, (b) you sell it separately as a product, or (c) the two codebases
